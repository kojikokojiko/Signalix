# アーキテクチャ概要

## システム設計思想

Signalix は **パイプライン中心** のシステムとして設計される。主要な関心事は、生の RSS コンテンツを
パーソナライズされた AI 強化済み記事フィードへと確実に変換することである。各コンポーネントは
明確に定義されたインターフェースで通信し、独立してデプロイ可能である。

## 各コンポーネントの説明

### フロントエンド (Next.js + TypeScript)
- SEO が必要なルート（ランディング、記事詳細）はサーバーサイドレンダリング。
- フィードは React Query でクライアントサイドデータフェッチ。
- ビジネスロジックを持たず、プレゼンテーションと API クライアントに専念する。

### API サーバー (Go)
- ステートレス REST API。
- 担当: 認証・ユーザーデータ・記事配信・フィード構成・ブックマーク・フィードバック。
- 重い AI 処理はインラインで行わず、ジョブキュー経由でワーカーに委譲する。
- レコメンドスコアはワーカーが事前計算した DB の値を読む。
- Redis の用途: セッション/トークンキャッシュ・フィードキャッシュ・レートリミットカウンタ。

### RSS インジェスションワーカー (Go)
- スケジュール駆動のフェッチャー（内部スケジューラまたは外部トリガー）。
- RSS XML をフェッチ・パースし、URL ハッシュで重複排除。
- 生記事を `articles` テーブルに `status=pending` で挿入。
- 処理キューにジョブを投入する。

### 記事処理ワーカー (Go + LLM API 呼び出し)
- `pending` 状態の記事を消費する。
- パイプラインステージ: 正規化 → 言語検出 → HTML クリーニング → 埋め込み生成 → 要約 → タグ抽出 → トレンドスコア計算。
- 各ステージはアトミック。失敗時は指数バックオフでリトライ。
- 完了時に記事を `processed` にマーク。

### レコメンドエンジン (Go)
- 新記事バッチ完了時またはスケジュールで起動。
- 埋め込み・ユーザー興味プロフィール・行動シグナルを読み込む。
- ランク付きの記事リストを `recommendation_logs` に書き込む。

### PostgreSQL
- すべての永続エンティティのプライマリデータストア。
- 埋め込みベクトル類似検索に pgvector 拡張を使用。

### Redis
- キャッシュレイヤー: フィード API レスポンス（TTL 5分）・ユーザーセッション。
- レートリミットカウンタ（スライディングウィンドウ）。
- Redis Streams でジョブキューを実装。

## デプロイトポロジー (AWS MVP)

```
Route53 → CloudFront → ALB
                         ├── ECS Task: API サーバー (複数レプリカ)
                         └── ECS Task: フロントエンド (Next.js SSR)

ECS Task: RSS ワーカー         (シングルレプリカ、スケジュール)
ECS Task: 処理ワーカー          (複数レプリカ、キュードリブン)
ECS Task: レコメンドワーカー    (シングルレプリカ、スケジュール)

RDS PostgreSQL (本番: Multi-AZ)
ElastiCache Redis
S3 (生コンテンツスナップショット、任意)
```

## 障害モードと対策

| 障害 | 対策 |
|------|------|
| RSS ソース到達不能 | バックオフリトライ。N 回失敗でソースを `degraded` にマーク |
| LLM API タイムアウト | バックオフリトライ。記事は `pending_ai` のまま残る。ジョブログにエラー記録 |
| DB 過負荷 | レコメンドクエリには読み取りレプリカを使用。コネクションプール上限設定 |
| ワーカーがジョブ処理中にクラッシュ | ジョブは成功時のみ ACK。未 ACK ジョブは再キューイング |
| 記事重複 | 挿入前に URL の SHA-256 ハッシュの一意制約でチェック |

## インターフェース規約

コンポーネント間のデータ交換はすべて以下を使用する:
- **REST JSON**: フロントエンド ↔ API サーバー間。
- **PostgreSQL テーブル**: ワーカーと API サーバー間の共有状態バス。
- **Redis Streams**: ジョブキューイベント。
- MVP ではワーカー間の直接 HTTP 通信は行わない。
